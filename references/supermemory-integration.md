## 一、Supermemory 核心原理

Supermemory 不是又一个向量数据库，而是定位为 **AI 的记忆层（Memory Layer）**。它的架构三层：

```
你的 AI 应用
    ↓
SuperMemory
    │
    ├── Memory Engine     事实抽取、时间追踪、矛盾消解、自动遗忘
    ├── User Profiles     静态事实 + 动态上下文
    ├── Hybrid Search     RAG + Memory 一体化查询
    ├── Connectors        Google Drive / Gmail / Notion / GitHub
    └── File Processing   PDF / 图像 / 视频 → 可搜索分块
```

## 二、融合点 1：矛盾消解 + 时间线追踪

### Supermemory 的做法

不同于传统记忆系统「新→旧」简单覆盖，Supermemory 能**追踪事实的时间线**：

```
用户说"我住在 NYC"   → 存为 timeline[0]: NYC
用户说"我刚搬到 SF" → 判断：这是对 timeline[0] 的更新
                     → 不删除旧记录，标记为"历史"
                     → 新增 timeline[1]: SF（当前有效）
                     → 查询时返回"住在 SF" + 标注"之前住 NYC"
```

**关键机制**：
- 不是简单地「新覆盖旧」，而是每条事实带时间戳
- 查询时自动判断哪条是最新的有效事实
- 历史事实不丢失，可供回溯
- **判断规则**：同主题、同实体类型、动词为状态变更（搬到/换到/改为）

### 记忆大师当前状态（v5.4.0）

- Reflect 阶段能检测矛盾 ✅
- 但处理方式是 `replace`（覆盖），丢失历史线索 ❌
- 没有时间线概念 ❌

### 融合方案（Phase 9 — v5.5.0+）

```python
# memory 格式扩展 — 带时间线的条目
格式: **[WF] 标题** — 内容 | Timeline:[{date: "2026-01", value: "NYC"}, {date: "2026-06", value: "SF"}] | Current: SF

# 自动学习 cron 检测到"搬到"类动词时：
# 1. 搜同主题旧条目
# 2. 追加 timeline 数组
# 3. 标记 current 为新值
# 4. 保留历史不删除
```

**实施步骤**：

| 步骤 | 内容 | 难度 |
|------|------|:----:|
| 1 | 识别"状态变更"类动词列表（搬到/改为/换成/升职/离职...） | 🟢 易 |
| 2 | 在 auto-learn cron 中集成 timeline 追加 | 🟡 中 |
| 3 | memory 格式扩展，支持 `Timeline:[]` + `Current:` 标注 | 🟡 中 |
| 4 | session_search 查询时优先返回 current | 🟢 易 |

## 三、融合点 2：静态 vs 动态画像分离

### Supermemory 的做法

```python
profile.static  → ["喜欢 TypeScript", "偏爱函数式风格"]  # 长期不变
profile.dynamic → ["正在开发 API 集成", "当前项目中"]     # 随时更新
```

**为什么重要**：
- 静态画像不需要频繁刷新，每次对话都要加载
- 动态画像可以更短的 TTL，自动过期
- 分开后，自动化系统知道哪些需要定期更新、哪些不用动

### 记忆大师当前状态（v5.4.0）

- `memory(target='user')` 是一个池子，全部混合
- 没有 static/dynamic 标记

### 融合方案（Phase 10 — v5.5.0+）

```yaml
# user profile 格式扩展
条目格式: **[类型] 内容** | TTL:YYYY-MM-DD | Layer:static|dynamic

# 分层规则
static:  姓名、职业、长期偏好、人格特征、固定习惯
dynamic: 当前项目、当前工作状态、近期计划、临时偏好

# TTL 默认值
static:  不设TTL 或 1年
dynamic: 30天 或 按项目周期

# 审计时的处理
static:  不动（除非用户明确修改）
dynamic: 每次审计检查时效性，过期自动标记
```

**实施步骤**：

| 步骤 | 内容 | 难度 |
|------|------|:----:|
| 1 | memory 格式扩展，支持 `Layer:static\|dynamic` 标注 | 🟢 易 |
| 2 | 自动学习 cron 写入时自动判断 static/dynamic | 🟡 中 |
| 3 | 每日审计区分对待 static/dynamic | 🟢 易 |
| 4 | dynamic 条目缩短默认 TTL 为 30天 | 🟢 易 |

## 四、融合点 3：自动遗忘增强（日期敏感信息）

### Supermemory 的做法

```
用户说"我明天有考试"
→ 检测到时间敏感词（明天/下周/月底...）
→ 自动标记短 TTL（考完试第二天过期）
→ 日期过后不再出现在 profile 中
→ 不会因为"考试过期了还存着"导致画像混乱
```

### 记忆大师当前状态（v5.4.0）

- TTL 是手动标注的（`TTL:YYYY-MM-DD`）
- 没有自动检测时间敏感词的机制

### 融合方案（Phase 9 — v5.5.0+，与矛盾消解同批）

**自动学习 cron 新增检测**：

```python
# 检测条件
TIME_SENSITIVE_PATTERNS = [
    r"明[天日年]",       # 明天/明日/明年
    r"下[周月年]",       # 下周/下月/明年
    r"[这今][周月天]",    # 这周/本月/今天
    r"周[一二三四五六日]", # 周一到周日
    r"\d+月\d+日",       # 6月20日
    r"(月底|月初|年末)",  # 月底/月初/年末
]

# 处理：
# 1. 检测到时间敏感词 → 计算过期日期
# 2. 自动追加 TTL
# 3. 在 memory 中标记为 dynamic（易变）
```

## 五、融合点 4：Memory ≠ RAG 概念强化

### Supermemory 的核心区分

> **RAG**：检索知识库文档片段 → 无状态、对所有人结果相同
> **Memory**：提取并追踪关于用户的事实 → 理解优先级与时效性

**区别表**：

| 维度 | RAG | Memory |
|------|-----|--------|
| 对象 | 文档/知识库 | 用户特定事实 |
| 状态 | 无状态 | 有状态（跨会话） |
| 个性化 | 所有人相同 | 每人不同 |
| 时效 | 静态文档 | 动态更新 |
| 冲突 | 无矛盾（都是文档） | 有矛盾需消解 |
| 优先级 | 无 | 有时效权重 |

### 在记忆大师中的映射

| 概念 | 我们的实现 |
|------|-----------|
| Memory | Layer② 用户记忆 + Layer① 自动学习 |
| RAG | session_search（检索历史会话） |
| 混合查询 | memory-vector.py 3通道RRF（向量+关键词+实体） |

**强化点**：在记忆大师文档中明确写出这个区分，帮助理解为什么记忆不能简单扔进向量库完事。
